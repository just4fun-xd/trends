# quantlab

Чистая пересборка исследовательского фреймворка систематических
стратегий (наследник EMA1221). Переписано с нуля, чтобы вычистить
накопленный дрейф старой архитектуры и заложить прочный фундамент под
мат-модели.

## Зачем переписано

Старая архитектура (`engine.py` + `strategies.py` + `main.py`) наращивалась
инкрементально, и накопились неточности, которые тянулись дальше и трудно
отлавливались. Главная — **close-only контракт**: стратегии получали
только цену закрытия, а Дончиану и ATR-стопу нужны реальные high/low.
Пробой по синтетическому хаю — не тот пробой. Здесь это исправлено на
уровне контракта данных.

## Три архитектурных решения

**1. Контракт `Bars`, не широкий DataFrame.** Стратегия получает
типизированный `Bars` (frozen dataclass: open/high/low/close/volume +
bars_per_year). Узкий контракт: нельзя случайно залезть в будущие строки
или посторонние колонки — это перекрывает look-ahead мимо единственного
`shift(1)` в движке. Старый close-only — частный случай `Bars.from_close()`.

**2. Единый `DataSource`, сменные бэкенды.** yfinance и Databento
реализуют один интерфейс и возвращают одинаковый `Bars`. Нормализация
(сплющить MultiIndex, UTC, обрезка `end` включительно) — ОДИН раз в
базовом классе, а не в каждом бэкенде. `interval` — параметр источника,
поэтому H4/интрадей ложатся в ту же архитектуру без спец-веток.
bars_per_year берётся из `Bars`, аннуализация волатильности корректна на
любом таймфрейме (252 дневные, ~1512 H4).

**3. Разрез по слою доверия.** Ядро + все пройденные стратегии
портированы и воспроизводят задокументированное поведение. Все
непройденные мат-модели — **интерфейсные заглушки, честно бросающие
`NotImplementedError`**, без выдумывания результатов.

## Databento: офлайн, вне горячего пути

Бэктест читает **локальные parquet-панели** (`DatabentoSource`) —
воспроизводимо, офлайн, не жжёт квоту. Живой API-pull вынесен в отдельный
ручной скрипт `scripts/fetch_databento.py`, запускаемый при расширении
корзины (7 → 20-30 инструментов). Схема панелей:
`panel_{open,high,low,close,volume,rollyield,native}.parquet` — по одному
инструменту на колонку, выровнены по union-календарю.

## Структура

```
core/        bars.py, engine.py, engine_portfolio.py, config.py
data/        source.py (abc), yfinance_source.py, databento_source.py
strategies/  ema.py, donchian.py, bollinger.py, ou.py, cross_sectional.py
regime/      detector.py (+HMM-заглушка), router.py
runners/     run_basket.py
scripts/     fetch_databento.py
tests/       test_core.py, test_strategies.py, test_regime.py
```

## Два движка

- `run_engine` — посерийные стратегии (одна позиция на инструмент).
- `run_portfolio` — кросс-секционные (матрица весов).
- Адаптер `positions_to_weights` мостит их для sanity-чека: на одном
  инструменте оба движка обязаны совпасть **до 0.0%** (проверяется тестом).

Архитектурная граница жёсткая: кросс-секционные стратегии нельзя гонять
посерийным циклом.

## Карта стратегий

| Стратегия | Класс | Статус |
|---|---|---|
| `ema_ensemble_voltarget` | equity | ✅ Чемпион акций (DD −21%, единственный EMA проходящий DD<40%) |
| `donchian_est_macd_4step_take` | commodity | ✅ Чемпион сырья (+5.2% / −12.1% / 15 из 19) |
| `bollinger_rsi` | range | ✅ Baseline, комплементарна тренду (профит в боковые годы) |
| `ou_zscore` | range | ⚠️ Математика валидна, НЕ автономна — RANGE-нога роутера |
| `ema_ensemble_long_short` | closed | ❌ Шорт на акциях = бета, не альфа |
| `donchian_breakout_ls` | closed | ❌ L/S на сырье (−71%) |

провалов с механистическим объяснением.

## Regime Router

Архитектурный ответ на «сотни стратегий, переключающихся по режимам
рынка» — реализованный через математику детектора, не ручное
переключение. `regime_router` мягко смешивает Donchian (TREND) / OU
(RANGE) / кэш (CRISIS) по вероятностям от `RegimeDetector`. Работает уже
сейчас с любым детектором; станет полноценным с HMM (трек 2.2). Под
`AlwaysTrendDetector` вырождается в чистый Donchian — проверяется тестом.

## Запуск

```bash
pip install -r requirements.txt

# Тесты (синтетические режимы — проверяют механику).
python -m tests.test_core
python -m tests.test_strategies
python -m tests.test_regime

# Прогон стратегии по корзине (нужна сеть для yfinance).
python -m runners.run_basket --strategy champion --basket commodity \
    --start 2021-01-01 --end 2026-01-01

# Выгрузка Databento (при расширении корзины).
export DATABENTO_API_KEY=...
python -m scripts.fetch_databento --symbols CL NG GC SI HG ZW ZC \
    --start 2015-01-01 --end 2025-01-01

# Демо-панели для теста Databento-пути без API-ключа.
python -m scripts.fetch_databento --demo
```

## Принципы

- Edge должен быть в сигнале, подходить инструменту, выживать после
  издержек.
- DD < 40% — жёсткий лимит на каждой стратегии.
- Медиана предпочтительнее среднего; выбросы (TSLA/NVDA) исключаются из
  статистики корзины.
- 5 лет — достаточная глубина бэктеста.
- Провалы документируются с механизмом, не хоронятся.
- Не выдумывать результаты для непройденных треков.

## Замечание о тестах

Живые данные Yahoo недоступны в изолированной среде сборки (хост не в
allowlist), поэтому тесты проверяют **механику на синтетических рядах**
с заданной структурой (трендовый / разворотный / боковой / OU). Это
строже сверки абсолютных чисел: убеждаемся, что пирамида набирается на
тренде, turtle-стоп выбивает на развороте, mean-reversion ловит боковик,
роутер точно совпадает с Дончианом под always-trend. Локально те же
стратегии на реальных данных воспроизводят числа из старых `.md`.
