# quantlab

Чистая пересборка исследовательского фреймворка систематических
стратегий (наследник EMA1221). Переписано с нуля, чтобы вычистить
накопленный дрейф старой архитектуры и заложить прочный фундамент под
мат-модели.

## Зачем переписано

Старая архитектура (`engine.py` + `strategies.py` + `main.py`) наращивалась
инкрементально, и накопились неточности, которые тянулись дальше и трудно
отлавливались. Главная — **close-only контракт**: стратегии получали
только цену закрытия, а Дончиану и ATR-стопу нужны реальные high/low.
Пробой по синтетическому хаю — не тот пробой. Здесь это исправлено на
уровне контракта данных.

## Аудит 2026-07

Пакет прошёл разбор внешнего ревью (6 замечаний) + самоаудит (8 находок).
Полный протокол с вердиктами — `docs/AUDIT_2026-07.md`. Ключевые правки
(в коде помечены «аудит 2026-07»):

- **Drift-издержки** в обоих движках: оборот считается против
  задрейфовавшего веса, а не `diff()` целевых (общая `drift_turnover`).
- **Buffer 10% в vol targeting**: размер не дёргается на микро-сдвигах
  волатильности (смен размера 470 → 35 на тесте).
- **One-shot take-profit + заморозка пирамиды** в 4step-ядре: устранены
  «пулемётный» тейк и пила тейк↔докупка.
- **Turtle-стоп подтягивается** за докупками (был прибит к entry).
- **Риск-триггеры по low, тейк по high**; входы close-confirmed.
- **Риск-паритет роутера**: сырые ноги + единый vol-target на итог
  (OU-нога больше не лупит ±1.0 против ужатого Дончиана).
- **NaN-устойчивый z-score**: std→NaN держит состояние (не фантомный
  выход и не ложный сигнал от 1e-9-подпорки).
- **Месячный ребаланс** кросс-секции (`rebalance_every=21`).
- Wilder RSI (канон), честное поле `bars_per_year` в результатах
  (pandas 3.0 CoW терял магический атрибут), tz-фикс интрадея,
  дедуп склейки в fetch_databento, тикер Zinc ZN=F (это T-Note!)
  заменён на Platinum PL=F.

**⚠️ Семантика champion уточнена** — эталонные числа старого кода
(+5.2% / −12.1% / 15 из 19) подлежат обязательной локальной
ре-валидации на реальных данных до доклада. Wilder RSI аналогично
меняет поведение BB+RSI.

## Три архитектурных решения

**1. Контракт `Bars`, не широкий DataFrame.** Стратегия получает
типизированный `Bars` (frozen dataclass: open/high/low/close/volume +
bars_per_year). Узкий контракт: нельзя случайно залезть в будущие строки
или посторонние колонки — это перекрывает look-ahead мимо единственного
`shift(1)` в движке. Старый close-only — частный случай `Bars.from_close()`.

**2. Единый `DataSource`, сменные бэкенды.** yfinance и Databento
реализуют один интерфейс и возвращают одинаковый `Bars`. Нормализация
(сплющить MultiIndex, UTC, обрезка `end` включительно) — ОДИН раз в
базовом классе, а не в каждом бэкенде. `interval` — параметр источника,
поэтому H4/интрадей ложатся в ту же архитектуру без спец-веток.
bars_per_year берётся из `Bars`, аннуализация волатильности корректна на
любом таймфрейме (252 дневные, ~1512 H4).

**3. Разрез по слою доверия.** Ядро + все пройденные стратегии
портированы и воспроизводят задокументированное поведение. Все
непройденные мат-модели — **интерфейсные заглушки, честно бросающие
`NotImplementedError`**, без выдумывания результатов.

## Databento: офлайн, вне горячего пути

Бэктест читает **локальные parquet-панели** (`DatabentoSource`) —
воспроизводимо, офлайн, не жжёт квоту. Живой API-pull вынесен в два
ручных скрипта — `scripts/fetch_databento_futures.py` (GLBX.MDP3,
M1/M2 continuous + carry) и `scripts/fetch_databento_equities.py`
(EQUS.SUMMARY) — оба переиспользуют общий тракт сборки/ресемпла/записи
из `scripts/panels_common.py`. Запускаются вручную при расширении
корзины, не на каждом прогоне бэктеста.

**Устойчивая выгрузка (`scripts/fetch_resilient.py`).** H4-стрим по
широкой корзине — это долгие потоки, которые обрываются на
транзиентных сетевых ошибках (`Response ended prematurely`). Тракт:

- **Per-symbol кэш.** Каждый инструмент, едва выгрузившись, сразу
  пишется в `{out}/raw/{symbol}.parquet`. Обрыв на середине корзины
  больше не теряет уже оплаченные данные — повторный запуск той же
  команды докачивает только недостающее.
- **Годовые чанки.** Каждый continuous-поток режется по календарным
  годам, а не качается одним 5-летним стримом — короче поток, ниже
  риск обрыва, дешевле retry.
- **Retry с backoff** на транзиентных ошибках (обрыв стрима, таймаут,
  5xx); нерезолвнутые символы (не тот датасет/биржа) пропускаются без
  падения всего прогона.
- **M1 переживает падение M2.** Если second-month контракт не
  поднялся даже после ретраев, front-month всё равно сохраняется —
  carry для этого инструмента просто не считается, а не теряются цены.

Схема панелей: `panel_{open,high,low,close,volume,rollyield,
native}.parquet` — по одному инструменту на колонку, выровнены по
union-календарю. Futures и equities пишутся в отдельные каталоги
(`data/panels/futures`, `data/panels/equities`) — разные sleeve.

**Известные ограничения корзины (проверено эмпирически, 2026-07):**
softs — кофе (KC), какао (CC), сахар (SB) — торгуются на ICE, не на
CME/GLBX.MDP3, и не резолвятся этим трактом; для них нужен отдельный
ICE-датасет (не реализовано). Хлопок (CT) на проверенном ключе не
вернул данных вовсе. На H4 палладий (PA) и платина (PL) дают
патологически низкое покрытие (`native` < 15% — тонкая ликвидность
continuous-контракта, реземпл в основном синтетика) — **исключай их
из H4-прогонов флагом `--exclude PA,PL`**; на дневках оба валидны.
Проверяй покрытие новой корзины через `check_panel_coverage.py` перед
тем как доверять H4-числам.

## Два движка

- `run_engine` — посерийные стратегии (одна позиция на инструмент).
- `run_portfolio` — кросс-секционные (матрица весов).
- Адаптер `positions_to_weights` мостит их для sanity-чека: на одном

## Балласт — свойство пары (актив × стратегия)

Вывод 2026-07f, полученный через `run_instrument_contribution`
(leave-one-out по инструментам): актив не бывает «хорошим» или «плохим»
сам по себе — он подходит или не подходит КОНКРЕТНОЙ стратегии.
Один и тот же инструмент под разными ногами:

| актив | тренд (donchian/kama) | MR (keltner/stops) | тип |
|---|---|---|---|
| CL, GC, ZS | держать | балласт в MR-стопах | трендовые |
| NG | балласт | **лучший держатель** в mr_kelt | реверсионный |
| PL | балласт во всех тренд-моделях | нейтрал в MR | не-трендовый |
| ZL, PA | нейтрал | балласт в MR-стопах | плохи для MR-стопов |
| GC, SI | держать | держать | сильные везде |

Практический смысл: NG и PL — не «плохой газ» и «плохая платина», а
не-трендовые активы, которые в тренд-ноге разбавляют сигнал, а в
MR-ноге работают. Ответ — не удалять их, а направлять в правильную
ногу. Это эмпирическое обоснование Hurst-аллокатора (вес тренд/MR по
показателю Хёрста инструмента), который стал приоритетом №1 в
`docs/ROADMAP_2026-07f.md`.

## rf в Sharpe: где вычитать, где нет

Sharpe считается двумя способами, и путать их — ошибка:

- **gross** (rf=0): инвариантен к масштабу позиции. Единственно
  корректен для сравнения стратегий между собой и для оценки отдельных
  ног/активов (у них разная реализованная вола; excess-Sharpe
  искусственно штрафует низковольные).
- **excess** (rf вычтен): осмыслен ТОЛЬКО на уровне полностью
  проинвестированного капитала (портфель/комбо). На уровне одной ноги
  даёт абсурд — вычет полной rf из актива с волой 3% превращает
  плюсовой P&L в Sharpe −2.

Для **фьючерсного счёта** excess-Sharpe = gross Sharpe: обеспечение
лежит в T-bills и зарабатывает rf само, поэтому excess счёта над
T-bills = чистый P&L стратегии. Вычитать rf из P&L фьючерсной
стратегии — двойной счёт. `run_sleeves`/`run_basket` печатают обе
рамки (cash-счёт и фьючерсный) + строку «доходность счёта = rf +
стратегия» (portable alpha).

## Regime Router

Архитектурный ответ на «сотни стратегий, переключающихся по режимам
рынка» — реализованный через математику детектора, не ручное
переключение. `regime_router` мягко смешивает Donchian (TREND) / OU
(RANGE) / кэш (CRISIS) по вероятностям от `RegimeDetector`. Работает уже
сейчас с любым детектором; станет полноценным с HMM (трек 2.2). Под
`AlwaysTrendDetector` вырождается в чистый Donchian — проверяется тестом.

## Запуск

```bash
pip install -r requirements.txt

# Тесты (синтетические режимы — проверяют механику).
python -m tests.test_core
python -m tests.test_strategies
python -m tests.test_regime

# Прогон стратегии по корзине (нужна сеть для yfinance).
python -m runners.run_basket --strategy champion --basket commodity \
    --start 2021-01-01 --end 2026-01-01

# Годовая разбивка return/DD по каждому инструменту (--yearly):
python -m runners.run_basket --strategy champion --basket commodity \
    --start 2021-01-01 --end 2026-01-01 --yearly

# Выгрузка Databento: два sleeve, два скрипта, устойчивый per-symbol
# кэш (scripts/fetch_resilient.py). Обрыв на середине корзины не
# теряет уже скачанное — просто перезапусти ту же команду.
export DATABENTO_API_KEY=...
python -m scripts.fetch_databento_futures \
    --symbols CL NG GC SI HG ZW ZC ZS ZL ZM PA PL \
    --start 2015-01-01 --end 2025-01-01     # -> data/panels/futures
python -m scripts.fetch_databento_equities \
    --start 2015-01-01 --end 2025-01-01     # -> data/panels/equities
# Перед первой equities-выгрузкой проверь глубину истории:
python -m scripts.fetch_databento_equities --check-range

# Демо-панели без API-ключа (тест труб):
python -m scripts.fetch_databento_futures --demo
python -m scripts.fetch_databento_equities --demo

# Кросс-секционные стратегии (dual momentum, carry) — ОТДЕЛЬНЫЙ раннер
# (посерийный run_basket их не гоняет — архитектурная граница):
python -m runners.run_xs --strategy dual_regime --basket equity \
    --start 2021-01-01 --end 2026-01-01 --yearly

# Kalman-пары (research) — свой раннер (нужны ДВА инструмента):
python -m runners.run_pairs --a CL=F --b BZ=F --yearly

# Лаборатория mean-reversion: 10 вариантов bb_rsi (mr_*), любой можно
# обернуть vol-таргетингом флагом --vt:
python -m runners.run_basket --strategy mr_atr_stop --vt \
    --basket commodity --start 2021-01-01 --end 2026-01-01 --yearly

# Anchored walk-forward: тест устойчивости по годам (фикс. параметры,
# без рефита). Несколько стратегий сразу для сравнения; --matrix
# печатает return год × инструмент. РОБАСТНА = >=80% окон прибыльны,
# худшее окно > -15%. ⚠️ Вердикт сейчас построен на консистентности
# return по окнам, а НЕ на DD<40% по факту (max_dd внутри окна
# зануляется намеренно — для честного DD нужен полный run_basket
# --yearly на общем периоде).
python -m runners.run_walkforward --strategy bb_rsi_vt mr_atr_stop \
    --basket commodity --start 2021-01-01 --end 2026-01-01 --vt --matrix

# --exclude вырезает инструменты с плохим покрытием на конкретном
# таймфрейме (см. раздел H4 ниже про PA/PL):
python -m runners.run_walkforward --strategy bb_rsi_vt \
    --basket commodity --source databento --panel-dir data/panels_4h/futures \
    --interval 4h --exclude PA,PL --vt --matrix

# Sleeve-комбинация: портфель из нескольких стратегий/корзин,
# ключевая диагностика — корреляция дневных P&L между sleeve'ами
# (trend vs MR должны быть близки к нулю/отрицательны):
python -m runners.run_sleeves --sleeve champion:commodity \
    --sleeve bb_rsi:commodity:vt --start 2021-01-01 --end 2026-01-01 --parity

# Вклад ИНСТРУМЕНТОВ в портфель (leave-one-out — детектор балласта).
# Для каждого актива: solo return/DD/Sharpe + LOO-дельта портфельного
# Sharpe (выкинули актив -> портфель лучше = балласт, хуже = держать).
# Sharpe gross (rf=0): инвариантен к масштабу позиции, честно
# сравнивает активы с разной волой. Ключевой вывод 2026-07f: балласт —
# свойство ПАРЫ (актив × стратегия), не актива. NG балласт в тренде,
# но лучший держатель в MR; PL не-трендовый во всех тренд-стратегиях.
python -m runners.run_instrument_contribution --strategy donchian_vt \
    --basket commodity --source databento --panel-dir data/panels/futures \
    --start 2019-01-01 --vt --target-vol 0.20

# Bootstrap: CI на Sharpe и на РАЗНОСТЬ Sharpe двух стратегий (арбитр
# смены чемпиона). Сравнение стратегий — на gross (без --rf): excess
# масштабо-зависим и загрязняет сравнение при разной воле ног. --rf
# нужен только для абсолютного уровня на портфеле/комбо.
python -m runners.run_bootstrap --a kama --b donchian_vt \
    --basket commodity --source databento --panel-dir data/panels/futures \
    --start 2019-01-01 --vt
```

## Расчёт на H4-данных

Databento. Тракт качает часовые бары (`ohlcv-1h`) и ресемплит в 4H в

**Через yfinance (быстрая проверка, только последние ~2 года):**

```bash
python -m runners.run_basket --strategy champion --basket commodity \
    --interval 4h --start 2024-07-01 --end 2026-01-01
```

**Через Databento (полные 5 лет, честный прогон):**

```bash

#    Softs (KC/CC/SB — ICE, не GLBX) и CT сюда не резолвятся, не
#    включай их. Обрыв на середине — просто перезапусти команду.
python -m scripts.fetch_databento_futures --interval 4h \

# 2. Sanity-check границы ресемпла и фактический bars_per_year на
```

5 лет, 2026-07) фактическое число H4-баров в году оказалось ~1589, не
1512 (6×252) — Globex-сессия не идеально регулярна (воскресные
открытия, часовые сдвиги). Разница смещает Sharpe на ~2.5% при
константе. `check_h4_bars.py` печатает факт по инструменту; если
`core/bars.py` у тебя всё ещё берёт `bars_per_year` статической
таблицей по интервалу, а не из фактического индекса панели — сверяй
вручную для каждого нового инструмента перед тем как доверять Sharpe.

Дневные и H4 панели держатся в разных каталогах (`data/panels/` и
`data/panels_4h/`), чтобы не путаться.

## Принципы

- Edge должен быть в сигнале, подходить инструменту, выживать после
  издержек.
- DD < 40% — жёсткий лимит на каждой стратегии.
- Медиана предпочтительнее среднего; выбросы (TSLA/NVDA) исключаются из
  статистики корзины.
- 5 лет — достаточная глубина бэктеста.
- Провалы документируются с механизмом, не хоронятся.
- Не выдумывать результаты для непройденных треков.

## Замечание о тестах

Живые данные Yahoo недоступны в изолированной среде сборки (хост не в
allowlist), поэтому тесты проверяют **механику на синтетических рядах**
с заданной структурой (трендовый / разворотный / боковой / OU). Это
строже сверки абсолютных чисел: убеждаемся, что пирамида набирается на
тренде, turtle-стоп выбивает на развороте, mean-reversion ловит боковик,
роутер точно совпадает с Дончианом под always-trend. Локально стратегии
прогоняются на реальных данных; из-за уточнённой семантики (аудит
2026-07) числа СВЕРЯЮТСЯ со старыми `.md` и фиксируются как эталон v2,
а не воспроизводятся бит-в-бит.
